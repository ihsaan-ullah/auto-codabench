# Checks covered by `autocodabench validate`

## 1. Structural

| **Check name**                                      | **LLM?** | **Description**                                                                                                                                                                                                       | **How it was checked**                                                                                                     |
| --------------------------------------------------------- | -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| Bundle schema and file references                         | No             | The structural gate:`competition.yaml`parses, every referenced file exists, programs carry runnable metadata, leaderboard keys are written by the scoring program. (Codabench bundle schema (Yaml-Structure.md))          | Parses `competition.yaml`and checks every referenced file, program metadata, and leaderboard key against the Codabench schema. |
| Leaderboard has a key; column keys and indices are unique | No             | Each leaderboard must carry a key, and its columns must have unique keys and unique indices — collisions silently drop or overwrite a ranked column on the platform. (Pavão et al. (Ch. 11); Codabench Yaml-Structure.md) | Checks each leaderboard declares a key and that its column keys and indices are unique.                                          |

## 2. Executable

| **Check name**                                 | **LLM?** | **Description**                                                                                                           | **How it was checked**                                                                             |
| ---------------------------------------------------- | -------------- | ------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Baseline runs end-to-end through scoring (in Docker) | No             | Run the bundle's own baseline through the scoring pipeline, in Docker. (Pavão et al. (Ch. 5, Ch. 11))                          | Runs the baseline through ingestion+scoring in the declared Docker image and checks a score is produced. |

| Starting-kit notebook executes cleanly (in Docker)   | No             | Execute the starting-kit notebook end-to-end inside the bundle's image. (Pavão et al. (Ch. 5, Ch. 13))                         | Executes the starting-kit notebook end-to-end in the declared Docker image.                              |
| Worker docker image pinned                           | No             | Silent dependency drift breaks reproducibility — pin the image. (Pavão et al. (Ch. 11))                                       | Reads `docker_image`and checks it carries an explicit (non-:latest) version tag.                       |

## 3. Methodological

| **Check name**                                             | **LLM?** | **Description**                                                                                                                                                                                                                                            | **How it was checked**                                                                              |
| ---------------------------------------------------------------- | -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| Baseline range (trivial→SOTA) documented (LLM-judged, advisory) | Yes            | Do the pages document a*range*of baselines — from a trivial bound to a competent/state-of-the-art method — and a meaningful gap between them? A single baseline cannot show the task has headroom wort… (Pavão et al. (Ch. 5))                             | An LLM checks the pages describe a range of baselines (trivial→SOTA) and the performance gap.            |
| Metric choice is justified (LLM-judged, advisory)                | Yes            | Do the pages justify*why*the chosen metric measures success on this task, not merely name it? A metric that is asserted but not motivated can fail to reward the behaviour the challenge cares about. (Pavão et al. (Ch. 4))                                  | An LLM checks the pages justify why the metric assesses the task, beyond naming it.                       |
| Score-difference significance addressed (LLM-judged, advisory)   | Yes            | Do the pages account for how meaningful score differences between participants are — error bars, significance, or noise? Minor for most challenges; flagged only when the ranking leans on fine distinct… (Pavão et al. (Ch. 4))                              | An LLM checks whether score-difference significance / error bars are addressed when the ranking needs it. |
| Task difficulty is calibrated (LLM-judged, advisory)             | Yes            | Do the pages argue the task is challenging but solvable — neither trivial nor impossible? Calibrating difficulty is a core design responsibility; the measured baseline gap is the evidence. (Pavão et al. (Ch. 5))                                            | An LLM checks the pages argue the task is challenging but solvable (e.g. a baseline-to-headroom gap).     |
| Daily submission cap on development phases                       | No             | Uncapped development submissions invite brute-force leaderboard overfitting; 5–10/day is the typical guard. (Pavão et al. (Ch. 5))                                                                                                                             | Reads `max_submissions_per_day`on each development phase.                                               |
| Development + final phase structure                              | No             | Single-phase competitions overfit the public leaderboard. (Pavão et al. (Ch. 5, Ch. 11))                                                                                                                                                                        | Counts the phases declared in `competition.yaml`.                                                       |
| Development phase ≥ 40 days                                     | No             | A development phase shorter than ~40 days only reaches people who were already working on the problem. (Pavão et al. (Ch. 13))                                                                                                                                  | Parses the first phase's start/end dates and computes the span in days.                                   |
| Final phase closes (review window exists)                        | No             | The final phase must have an explicit end date: an open-ended final phase never closes, so there is no defined point at which organizers begin reviewing winners and analysing results. (Pavão et al. (Ch. 5))                                                  | Checks the final phase declares an end date (a defined close after which organizer review can begin).     |
| Final phase total-submission limit ≤ 3                          | No             | The final phase exists to close the overfit loophole: 1–3 total submissions on the never-seen private set. (Pavão et al. (Ch. 5))                                                                                                                              | Reads `max_submissions`on the final phase.                                                              |
| Phase dates are coherent and ordered                             | No             | Phase dates must be internally coherent: each phase ends after it starts, and phases run in chronological order. Scrambled or inverted phase dates silently break the competition timeline on the platfo… (Pavão et al. (Ch. 11); Codabench Yaml-Structure.md) | Parses every phase's start/end and checks each end ≥ start and phase starts are non-decreasing.          |
| Test set sized for the anticipated error rate (100/E)            | No             | The 100/E rule: to resolve top systems at anticipated error rate E, you need roughly 100/E test examples. (Pavão et al. (Ch. 4))                                                                                                                                | Compares the reference/test row count against 100 / anticipated-error-rate.                               |
